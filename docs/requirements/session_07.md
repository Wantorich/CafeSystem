# 요구사항: 카페 주문 시스템 — 결제 모듈

## 배경

카페 시스템에 결제 기능을 추가하기로 했다.
현재 개발팀은 카드 결제만 지원하는 간단한 구현을 빠르게 만들었다.

```java
public class OrderService {

    public void processPayment(Order order) {
        CardPayment payment = new CardPayment();  // 구체 클래스에 직접 의존
        int amount = order.calculateTotalPrice().getMoney();
        payment.pay(amount);
    }
}
```

출시 후 요구사항이 추가되었다.
- 카카오페이 결제도 지원해야 한다.
- 테스트 환경에서는 실제 결제가 일어나면 안 된다.
- 나중에 네이버페이, 토스 등 다른 결제 수단이 추가될 수 있다.

현재 구조에서 카카오페이를 추가하려면 `OrderService`를 수정해야 한다.
테스트 환경을 위해서도 `OrderService`를 수정해야 한다.

---

## 기능 요구사항

- FR-01: 카드 결제를 처리할 수 있다.
- FR-02: 카카오페이 결제를 처리할 수 있다.
- FR-03: 테스트용 가짜 결제(실제 결제 없이 성공 처리)를 지원한다.
- FR-04: 새로운 결제 수단이 추가될 때 `OrderService`를 수정하지 않아야 한다.
- FR-05: 주문의 최종 결제 금액은 할인이 적용된 총액이다.

---

## 제약 조건

- `OrderService`는 구체적인 결제 구현체를 직접 생성하지 않는다.
- 순수 Java로만 구현한다.

---

## 검증 시나리오

**시나리오 1 — 카드 결제**
- 아메리카노(4,500원) 2잔 주문, 멤버십 할인 적용 → 카드로 8,100원 결제된다.

**시나리오 2 — 카카오페이 결제**
- 동일한 주문을 카카오페이로 8,100원 결제된다.

**시나리오 3 — 테스트 환경**
- 가짜 결제 수단으로 실제 외부 호출 없이 결제가 성공 처리된다.

**시나리오 4 — 새 결제 수단 추가**
- 네이버페이 결제를 추가할 때 `OrderService` 코드를 수정하지 않아도 된다.

## 구현 시작 전 스스로 답해볼 것

1. 현재 배경 코드의 문제가 무엇인가? `new CardPayment()`가 `OrderService` 안에 있으면 무엇이 어려워지는가?
2. `OrderService`가 구체 클래스를 모르게 하려면 어떻게 해야 하는가? 누가 구체 클래스를 결정하고 전달해야 하는가?
3. 테스트 환경에서 가짜 결제 수단을 끼워 넣으려면 지금 구조에서 어떻게 해야 하는가?
